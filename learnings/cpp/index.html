References & Pointers

Aliases, referents, pass-by-ref, pointer vs reference
+

What is a Reference?

A reference is an alternative name (alias) for an existing object — the referent. It is not a copy, not a new object; it is the referent itself accessed under a different name.

int i = 10;
int &j = i;   // j is a reference (alias) to i — i is the referent
j = 11;        // both i and j are now 11 — same memory location

Two Forms of References

  • Local reference — int &j = i; — alias within the same scope
  • Parameter reference — void swap(int &i, int &j) — alias bound to an argument

Key Rules of References

  • A reference must be initialised at declaration — you cannot create an unbound reference
  • A reference cannot be rebound to a different referent after initialisation
  • Taking the address of a reference gives the same address as the referent
  • Assigning a value to a reference changes the referent's value
  • int& const x — makes no sense; references are already implicitly const-bound

Pointers vs References

When to Use Each

Prefer references when there is no need to reseat the value or support null. Use pointers when the target may change or may be null.

Returning a Reference — Method Chaining

Returning *this from a member function allows method chaining — each call returns the same object so the next call can be applied to it immediately.

// cout << x << y; works because:
// cout.operator<<(x) returns cout (by reference), then .operator<<(y) is called

C++ OOP Concepts

Encapsulation, abstraction, polymorphism, inheritance, classes
+

The Four Pillars

Encapsulation

Wrapping data members and the functions that operate on them into a single unit (the class). Access levels control who can read or modify the data — enforcing data security through information hiding.

class Person {
private:
    char* name;  // hidden — only accessible via public methods
    int age;
public:
    void setName(const char*);
    char* getName() const;
};

Data Abstraction

Representing only the essential information while hiding implementation details. The user of a class sees what it does, not how.

Example: vector.size() tells you the count of elements. You don't need to know how the count is tracked internally — that's abstraction.

Polymorphism — "Many Forms"

TypeResolved atMechanism
Static (compile-time)Compile timeFunction overloading, operator overloading
Dynamic (runtime)RuntimeVirtual functions, function overriding

Example: A Shape class with a draw() function behaves differently depending on whether the object is a Circle, Triangle or Rectangle.

Inheritance

The mechanism of deriving a new class from an existing one — reusing and extending behaviour without modification.

  • Represents an "is-a" relationship (TV is a Device)
  • Types: Single, Multiple, Multi-level
  • Produces a hierarchical relationship — general at the top, specialised at the bottom
  • Promotes code reuse and extensibility

Classes and Objects

  • Class — a blueprint / template defining a new type and scope. Once defined, no new members can be added.
  • Object — a runtime instance of a class; storage is allocated only when an object is defined, not when the class is defined.
  • struct keyword — members before the first access label are public by default
  • class keyword — members before the first access label are private by default
Size of an empty class: 1 byte — not zero. The C++ standard requires that no two distinct objects share the same memory address. To guarantee this, the compiler adds a dummy byte to any class with no data members and no virtual functions.

this Pointer

A hidden pointer inside every non-static member function pointing to the object on which the function was called. Used to return the current object by reference:

example& example::memfunc() {
    // ... do work ...
    return *this;  // enables method chaining
}

Classes, Constructors & Copy Control

Constructors, destructors, copy constructor, operator=, placement new
+

Constructors

A special member function with the same name as the class, called automatically by the compiler whenever an object is created. Used to initialise data members to sensible values.

  • Constructors can be overloaded — multiple constructors with different signatures
  • A default constructor takes no arguments (or all arguments have default values)
  • Should not call another constructor as a primitive — use init() helper or default parameter values instead
class Example {
    int a;
    std::string data;
public:
    Example() : a(0) {}               // default constructor
    Example(int x) : a(x) {}           // int overload
    Example(const char* s) : a(0), data(s) {} // string overload
};

Destructors

Called automatically when an object goes out of scope or is explicitly deleted. Used to release resources (heap memory, sockets, file handles).

  • Derived class destructors are called first, then the base class destructor — reverse construction order
  • Never call destructors explicitly unless using placement new (the only exception)
  • Do not call virtual functions inside constructors or destructors — the derived part may not yet exist or may already be destroyed
Virtual destructors: Always declare the base class destructor as virtual when the class is designed to be inherited. Without it, deleting a derived object through a base pointer results in undefined behaviour — the derived destructor is never called.
class Base {
public:
    virtual ~Base() {}  // virtual — ensures derived destructors are called
};
Base* p = new Derived;
delete p;  // calls ~Derived() first, then ~Base() ✓

Copy Constructor

Invoked when an object is copied — passed by value, returned by value, or explicitly copied. The compiler generates a default copy constructor that performs a shallow (bitwise) copy.

The pointer problem: If your class contains a raw pointer, the default copy constructor copies the pointer address — not the data it points to. Both objects then point to the same memory. When one destructor runs and frees that memory, the other object holds a dangling pointer — crash on next access or double-delete.

Solution: Write your own copy constructor to perform a deep copy:

test(const test& t) {
    val = new int(*(t.val));           // deep copy — allocate new memory
    str = new char[strlen(t.str)+1];
    strcpy(str, t.str);
}

Three situations where copy constructor is called:

  1. Explicitly copying an object: Test b = a;
  2. Passing an object by value to a function
  3. Returning an object by value from a function

Assignment Operator (operator=)

Called when assigning one existing object to another. Like the copy constructor, the default performs shallow (bitwise) copy — same dangling pointer risk.

Rules for a safe operator=:

  • Always return *this to enable chaining: a = b = c = d
  • Check for self-assignment first: if (this == &rhs) return *this;
  • Assign all data members — including the base class: call static_cast<Base&>(*this) = rhs;
  • Delete before allocating — free old memory before pointing to new memory
myString& operator=(const myString& rhs) {
    if (this == &rhs) return *this;  // self-assignment guard
    delete[] data;                    // free existing memory
    data = new char[strlen(rhs.data)+1];
    strcpy(data, rhs.data);
    return *this;
}

Copy Elision & NRVO

Compiler optimisations that eliminate redundant copy constructor and destructor calls. Named Return Value Optimisation (NRVO) avoids copying a locally constructed return value — the object is constructed directly in the caller's storage.

These optimisations make returning objects by value practical. Trust your compiler — don't pessimise by returning pointers just to "avoid copies".

Initialisation Lists & Inline Functions

Member init lists, ordering, inline vs macro
+

Constructor Initialisation Lists

Prefer initialisation over assignment in constructors. Initialisation is faster (one step), assignment is two steps (default-construct then assign). For const members and references it is mandatory — they cannot be assigned, only initialised.
// ✓ Initialisation list — one step, always correct
Example(int i) : a(i), b(i) { }

// ✗ Assignment in body — two steps, fails for const/ref members
Example(int i) { a = i; b = i; }

Rules for Initialisation Lists

  • Members are initialised in the order they are declared in the class — not the order in the init list. Always write them in declaration order to avoid confusion.
  • const and reference members must use init lists — they cannot be assigned
  • Static data members are never initialised in the constructor — they are initialised once at their definition outside the class
  • All data members and base class constructors should appear explicitly in the init list

Inline Functions

The inline keyword requests the compiler to insert the function's code at the call site instead of performing a real function call.

AspectMacro (#define)Inline Function
Type checking❌ None✅ Full argument type checking
Conversions❌ Not performed✅ Performed correctly
Debugging❌ Hard — expanded text✅ Works with debugger like a normal function
Increment side effects❌ ++i may run twice✅ Evaluated once, passed as value
// Macro — i incremented TWICE
#define myMacro(x) ((x>=0) ? x : -(x))
val = myMacro(++i);  // danger!

// Inline — i incremented once, value passed correctly
inline int myInline(int x) { return (x>=0) ? x : -(x); }
val = myInline(++i);  // safe ✓

Key Points About Inline

  • The compiler is not obligated to honour the inline keyword — recursive or very long functions will not be inlined
  • Best practice: put inline on the definition only (outside the class body)
  • A member function defined inside the class body is implicitly inline
  • Virtual functions can only be inlined when called on an actual object (not through a pointer/reference) — the compiler must know the exact type
  • main() must never be inlined

Pointer Const Declarations

Draw a mental vertical line through the asterisk: const left of * = the pointed-to data is const. const right of * = the pointer itself is const.

char* p✅ Yes✅ Yes
const char* p✅ Yes❌ No

const char* const p❌ No❌ No

Const Member Functions

A const member function (also called an inspector) guarantees it won't modify *this. A non-const member function (a mutator) may modify the object.

class Fred {
public:
    void inspect() const;  // inspector — won't change *this
    void mutate();          // mutator — may change *this
};

Fred const* p = &f;
p->inspect();  // ✓ OK
p->mutate();   // ✗ Error — p promises not to change *p

mutable Data Member

A mutable member can be modified even inside a const member function. Use for logically-irrelevant state like caches or call counters.

class Example {
    mutable int callCount;
public:
    int getData() const { ++callCount; return data; } // OK — mutable
};

Pass by Const Reference

Prefer pass-by-const-reference over pass-by-value for class types. Pass-by-value invokes the copy constructor (and all base class and member copy constructors) — potentially dozens of function calls for one argument. Pass-by-const-ref has zero overhead and also prevents the slicing problem.

Slicing problem: When a derived class object is passed as a base class object by value, the derived-class-specific members are silently stripped away. Passing by reference avoids this — the virtual dispatch still works.

// ✗ Slices derived features off
void print(University obj) { obj.display(); }

// ✓ Preserves polymorphism — correct virtual function is called
void print(const University& obj) { obj.display(); }

When NOT to Use Const

  • Do not declare const formal parameters that are passed by value — const is meaningless for the caller
  • Do not mark a function return type as const when returning by value — it prevents move optimisations

Static Members

Static data, static functions, inheritance, SIOF
+

Static Data Members

Associated with the class itself, not with individual objects. There is exactly one instance of a static data member regardless of how many objects of the class are created.

  • Initialised outside the class at their definition — not in a constructor init list
  • Accessed via ClassName::member, or through any object/pointer of that type
  • Also known as class data
class Counter {
public:
    static int count;  // declaration
    Counter() { count++; }
};
int Counter::count = 0;  // definition + initialisation (outside class)

Static Member Functions

Have no this pointer — they are not called on an object. Can only access other static members directly; to access non-static members, an object must be passed as a parameter.

  • Also known as class services / class methods
  • Cannot be virtual, const, or volatile
  • Similar to a friend function — no implicit this, can access non-public members

Static & Inheritance

If a base class defines a static member, there is one instance shared across all derived classes. Static members are not overridable — they are hidden (not overridden) in derived classes. A base class pointer to a derived object still calls the base class static function.

Static Initialisation Order Fiasco (SIOF): The order of initialisation of static objects across different translation units is undefined. Never rely on one translation unit's static being ready for another's. Use local static or lazy initialisation patterns to avoid this.

Analogy

Static member functions are like services provided by the factory itself, not by the products it creates. The factory (class) exists and can offer its services even before any product (object) is instantiated.

Friend Classes & Functions

Friendship rules, operator overloading, virtual friend idiom
+

What is a Friend?

A mechanism that grants a specific class or function access to the non-public members of the granting class. Friends are not members and are unaffected by access labels.

Three Rules of Friendship

  • Friendship is not transitive — a friend of a friend is not automatically a friend
  • Friendship is not inherited — derived classes do not inherit the friendship of their base class
  • Friendship is not virtual — friend functions do not bind dynamically (but the virtual friend function idiom provides this effect via a protected virtual helper)

Operator << and >> Overloading

These are overloaded as non-member friend functions rather than member functions because cout/cin (the ostream/istream object) must be on the left side of the operator — which is impossible as a member function.

class MyClass {
    int a;
public:
    friend std::ostream& operator<<(std::ostream& os, const MyClass& rhs);
    friend std::istream& operator>>(std::istream& is, MyClass& rhs);
};

std::ostream& operator<<(std::ostream& os, const MyClass& rhs) {
    return os << rhs.a;
}

Virtual Friend Function Idiom

Friend functions do not bind dynamically — but you can achieve the effect by having the friend call a protected virtual helper:

class Shape {
public:
    friend std::ostream& operator<<(std::ostream& os, const Shape& s) {
        s.print(os);  // calls the right derived class print() via virtual dispatch
        return os;
    }
protected:
    virtual void print(std::ostream&) const = 0;  // pure virtual
};

Virtual Functions & Inheritance

vtable, vptr, dynamic dispatch, abstract classes, derived classes
+

How Virtual Functions Work — vtable & vptr

When the compiler sees a virtual keyword, it builds a vtable (virtual function table) for that class — an array of function pointers, one per virtual function.

Construction Steps

  1. vtable built per class — one table per class type, shared across all objects
  2. vptr added to each object — a hidden pointer inside every object pointing to its class's vtable
  3. Constructor initialises vptr — the constructor sets vptr = &ClassName::vtable
// Derived vtable overlays base entries with overrides:
// Base vtable:    [&Base::func, &Base::func1, &Base::func2, &Base::func3]
// Derived vtable: [&Base::func, &Derived::func1, &Derived::func2, &Base::func3]

Base* bPtr = new Derived;
bPtr->func1();  // follows vptr → Derived vtable → calls Derived::func1 ✓

When Dynamic Binding Applies

  • Virtual functions called through a pointer or reference to a base class → resolved at runtime
  • Non-virtual functions → resolved at compile time based on the static type
  • Virtual functions called on an actual object (not pointer/ref) → can be inlined by the compiler

Important Rules

  • Any non-static member function other than constructors may be virtual
  • Static member functions cannot be virtual
  • Constructors cannot be virtual — the object doesn't fully exist when the constructor runs
  • Don't call virtual functions in constructors or destructors — the object is incomplete and the derived override won't be invoked
Virtual constructors don't exist — object creation and type are tightly coupled in C++. Use the Factory Method pattern to decouple object creation from type selection at runtime.

Abstract Classes & Pure Virtual Functions

A class containing at least one pure virtual function is an abstract class.

class Shape {
public:
    virtual void draw() = 0;  // pure virtual — must be overridden
    virtual ~Shape() {}        // virtual destructor
};
  • Cannot be instantiated — only used as a base
  • Derived classes must implement all pure virtual functions or they become abstract too
  • Can have non-abstract members alongside pure virtual ones

Default inheritance: class keyword defaults to private; struct keyword defaults to public.

References & Pointers

What is a reference and how does it differ from a pointer?
A reference is an alias for an existing object — it doesn't create a new object or copy. It must be initialised at declaration and cannot be rebound to a different object. A pointer is an independent object that stores a memory address and can be changed to point elsewhere or set to null. Prefer references when the target is guaranteed to exist and won't change.
Can a reference be made to refer to a different object after initialisation?
No. Once a reference is bound to a referent, it is bound permanently. Any assignment to the reference changes the value of the referent, not the binding. This is a fundamental difference from pointers.
Does int& const x make sense?
No. A reference is implicitly const-bound — it cannot be reseated. Adding const to the reference itself adds no meaning and is an error. const int& x (const applied to what is referenced) is meaningful and common.

OOP & Classes

What is the size of an empty class in C++?
1 byte — not zero. The C++ standard says no two distinct objects may share the same memory address. To guarantee unique addresses, the compiler inserts a dummy byte into any class with no data members and no virtual functions.
What is the difference between struct and class in C++?
The only technical difference is the default access level. In a struct, members before the first access label are public. In a class, they are private. Default inheritance is also public for struct and private for class.
What is the difference between abstract function and virtual function?
A virtual function has a body — it provides a default implementation that derived classes may optionally override. A pure virtual (abstract) function has no body (declared with = 0) and must be overridden in every concrete derived class. You cannot call base.Method() on an abstract function since there is no base implementation.

Constructors & Destructors

Why should the base class destructor be virtual?
When you delete a derived object through a base class pointer, the compiler decides which destructor to call based on the pointer type. Without virtual, only the base destructor runs — the derived destructor is skipped, leaking resources. With virtual, dynamic dispatch ensures the derived destructor runs first, then the base destructor.
When is a copy constructor invoked?
Three situations: (1) when an object is explicitly copied Test b = a;, (2) when an object is passed by value as a function argument, (3) when an object is returned by value from a function. The compiler generates a default shallow (bitwise) copy constructor — write your own for classes with raw pointer members.
Should one constructor call another constructor as a primitive?
No. Even though C++ allows it syntactically (before C++11 delegating constructors), calling a constructor inside another constructor creates a temporary object, not this. That temporary is initialised and then immediately destroyed, leaving the original object's members uninitialised. Better approaches: use default parameter values or a shared private init() helper method.
Why shouldn't you call virtual functions in a constructor or destructor?
During base class construction, the object's type is the base class type — the derived part hasn't been built yet. Calling a virtual function would invoke the base version, not the derived override. The same applies in reverse during destruction: the derived part is already destroyed when the base destructor runs, so calling virtual functions there would reach a destroyed derived object.
Why are initialisation lists preferred over assignment in constructors?
Initialisation is a single step — the member is constructed with the given value directly. Assignment is two steps — the member is first default-constructed (often expensively), then assigned a new value. Additionally, const and reference members can only be initialised, never assigned, so init lists are mandatory for them.

Const & Static

What is the difference between Fred const* p, Fred* const p, and Fred const* const p?
Fred const* p — pointer is changeable, but it cannot be used to change the Fred object. Fred* const p — pointer is constant (cannot point elsewhere) but the Fred object can be modified through it. Fred const* const p — both the pointer and the Fred object are constant.
How is a static member function similar to a friend function?
Both have no implicit this pointer parameter, and both can access the non-public members of the class. The key difference: a friend is an external function granted access, while a static member is part of the class itself and follows the class's access rules.

Virtual Functions & Polymorphism

How does dynamic dispatch work internally (vtable)?
The compiler creates a vtable (array of function pointers) for each class with virtual functions. Each object gets a hidden vptr pointing to its class's vtable. The constructor sets the vptr. When a virtual function is called through a pointer/reference, the runtime follows the vptr to the vtable and calls the correct function pointer — hence "runtime dispatch".
Can a static member function be virtual?
No. Static member functions have no this pointer, so they cannot participate in virtual dispatch (which requires an object's vptr). The compiler will reject virtual static void func().
What is the slicing problem and how do you prevent it?
When a derived class object is passed by value as a base class object, the derived-class-specific members are silently stripped away — "sliced off". The result is a pure base class object. Prevention: always pass class objects by reference (const Base& obj), not by value. This preserves the full object and virtual dispatch works correctly.
Why is passing by reference more efficient than passing by value for class objects?
Pass-by-value invokes the copy constructor of the object and all its member and base class copy constructors and destructors. For a class with inherited members and string fields, a single argument can trigger 6+ constructor and 6+ destructor calls. Pass-by-const-reference has zero copy overhead — just one pointer-sized value passed to the function.
